# Lucive 이메일 변수명 및 발송 트리거 명세서

기준일: `2026-04-22`  
목적: Klaviyo, Tally, Zapier 기준으로 메일 발송에 필요한 **공통 변수명**, **이벤트 트리거**, **조건 분기**를 한 문서로 정리한다.

---

## 1. 기본 원칙

- `Klaviyo`는 이메일 프로필과 발송 세그먼트의 기준이 된다.
- `Tally`는 스크리닝 응답의 원본이다.
- `Zapier`는 이벤트를 받아 Klaviyo 프로필 업데이트, 시트 갱신, 메일 발송을 수행한다.
- 연결 키는 기본적으로 `email`이다.

---

## 2. 메일별 발송 순서

메인 퍼널:

1. `00_kr_등록_확인.html`
2. `01_설문완료_로스터대기.html`
3. `02_알파_선발.html` 또는 `03_알파_대기자.html`
4. `04_알파_이용가이드_오픈전날.html`

이벤트 메일:

- `90_순위상승_알림.html`

EN/JP는 같은 순서를 따른다.

---

## 7. 메일별 필수 변수

## 7-1. `00 등록확인 + 추천 리더보드 안내`

필수:
- `user_name`
- `survey_url`
- `user_rank`
- `community_primary_url` 또는 시장별 채널 URL
- `unsubscribe_url`
- `privacy_url`

선택:
- `source`
- `market`

트리거:
- `registration_completed = true`
- `survey_completed = false`
- 최초 1회만 발송

## 7-2. `01 설문완료 / 로스터대기`

필수:
- `user_name`
- `referral_share_url`
- `user_rank`
- `community_primary_url`
- `selection_announce_date`

트리거:
- `survey_completed = true`
- `roster_status = waiting`
- 최초 1회만 발송

## 7-3. `02 알파 선발`

필수:
- `user_name`
- `invite_link`
- `alpha_start_date`
- `alpha_end_date`
- `guide_url`
- `bug_report_url`
- `community_primary_url`
- `user_rank`
- `referral_count`

트리거:
- `selection_status = selected`
- 최초 1회만 발송

## 7-4. `03 알파 대기자`

필수:
- `user_name`
- `next_wave_date`
- `user_rank`
- `referral_count`
- `community_primary_url`

선택:
- `waitlist_position`
- `dip_consolation_amount`

트리거:
- `selection_status = waitlist`
- 최초 1회만 발송

## 7-5. `04 알파 이용가이드 / 오픈 전날`

필수:
- `user_name`
- `guide_url`
- `bug_report_url`
- `alpha_start_date`

트리거:
- `selection_status = selected`
- `alpha_start_date - 1 day`
- `guide_sent = false`

## 7-6. `90 순위 상승 알림`

필수:
- `user_name`
- `user_rank`
- `referral_count`
- `next_goal_gap`
- `referral_share_url`

트리거:
- `rank_update_eligible = true`
- `user_rank` 값이 개선되었을 때
- 발송 빈도 제한 필요

## 8. Zapier 기준 이벤트 트리거

## 8-1. 등록 완료

이벤트명:
- `registration_completed`

입력 소스:
- 랜딩 페이지 폼 제출

Zapier 액션:
1. Klaviyo 프로필 생성/업데이트
2. `registration_completed = true`
3. `00` 메일 발송

## 8-2. 설문 완료

이벤트명:
- `survey_completed`

입력 소스:
- Tally submission webhook

Zapier 액션:
1. Klaviyo 프로필 업데이트
2. `survey_completed = true`
3. `roster_status = waiting`
4. 시트/Airtable 로스터 행 생성 또는 갱신
5. `01` 메일 발송

## 8-3. 선발 상태 변경

이벤트명:
- `selection_status_changed`

입력 소스:
- 운영 시트 또는 Airtable에서 상태 변경

Zapier 액션:
1. Klaviyo 프로필 업데이트
2. 조건 분기
   - `selected` → `02`
   - `waitlist` → `03`

## 8-4. 오픈 전날 가이드

이벤트명:
- `alpha_day_before`

입력 소스:
- Zapier Schedule

조건:
- `selection_status = selected`
- `guide_sent != true`

액션:
1. `04` 발송
2. `guide_sent = true`

## 8-5. 순위 상승

이벤트명:
- `rank_improved`

입력 소스:
- 추천 집계 시트 / DB

조건:
- 이전 `user_rank` > 현재 `user_rank`
- 최소 변화 폭 기준 필요
- 하루 1회 제한 권장

액션:
1. `user_rank`, `referral_count`, `next_goal_gap` 갱신
2. `90` 메일 발송

---

## 9. 추천 발송 제한 규칙

`90 순위상승`은 로직을 꼭 제한해야 한다.

권장 규칙:
- 하루 최대 1회
- 순위 개선 폭이 `10` 이상일 때만
- 또는 보상 구간 진입 시만
- 발송 후 `last_rank_email_at` 기록

추가 변수 권장:

| 변수명 | 타입 | 설명 |
|---|---|---|
| `last_rank_email_at` | datetime | 마지막 순위 메일 발송 시각 |
| `previous_rank` | number | 직전 순위 |
| `rank_delta` | number | 순위 상승 폭 |

---

## 10. 발송 트리거 요약표

| 메일 | 트리거 | 조건 | 중복 발송 |
|---|---|---|---|
| `00` | 등록 완료 | 설문 미완료 | 1회 |
| `01` | 설문 완료 | 로스터 대기 | 1회 |
| `02` | 선발 상태 selected | 선발자 | 1회 |
| `03` | 선발 상태 waitlist | 대기자 | 1회 |
| `04` | 오픈 전날 스케줄 | 선발자, 가이드 미발송 | 1회 |
| `90` | 순위 상승 이벤트 | 랭크 상승, 빈도 제한 충족 | 반복 가능 |

## 11. 추천 구현 순서

1. Klaviyo 프로필 속성 이름 확정
2. Tally hidden fields / response fields 이름 확정
3. 운영 시트 컬럼명을 위 변수명과 맞춤
4. Zapier에서 이벤트별 Zap 생성
5. 테스트용 더미 유저 3명으로 E2E 검증

---

## 12. 한 줄 정리

운영 연결의 핵심은 `email` 하나로 `Klaviyo 프로필`, `Tally 응답`, `로스터 상태`, `추천 순위`를 묶고, 메일은 `등록 → 설문 → 선발/대기 → 오픈 전날 → 순위상승` 순서로 자동 분기시키는 것이다.
