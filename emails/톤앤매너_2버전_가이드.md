## 1. 결론 먼저

Lucive 이메일은 아래 2개 버전으로 운용하는 것이 가장 자연스럽다.

### Version A. 제품 톤앤매너
- 성격: `정확함`, `안내 중심`, `운영 신뢰`, `제품 가치 명료화`
- 적합한 메일:
  - 설문 유도
  - 설문 완료 / 대기 등록
  - 선발 축하
  - 제품 이용 가이드
  - 버그 리포팅 가이드
  - 미선발 안내
- 핵심 느낌:
  - “무엇이 완료되었는지”
  - “다음에 무엇을 해야 하는지”
  - “무엇을 기대하면 되는지”
  - “Lucive가 기존 서비스와 무엇이 다른지”

### Version B. 랜딩 페이지 톤앤매너
- 성격: `장면 중심`, `서사적`, `2인칭 몰입`, `감정적 후크`
- 적합한 메일:
  - 웰컴 메일
  - 추천 참여 유도
  - 리더보드 / 순위 상승 메일
  - 커뮤니티 기대감 고조 메일
- 핵심 느낌:
  - “이제 당신 차례다”
  - “당신의 선택이 다음 장면을 연다”
  - “누군가가 당신을 기다리고 있다”
  - “루시브의 세계가 이미 움직이기 시작했다”

실무 추천:
- `운영/절차 메일`은 **제품 톤**
- `참여/몰입/공유 메일`은 **랜딩 톤**

## 2. 두 톤의 원형

## 2-1. 제품 톤앤매너 정의

현재 프로덕트와 운영 문서에서 읽히는 톤은 아래에 가깝다.

- 기억, 설정, 관계 지속성 같은 `제품 약속`을 분명히 말한다
- 과장보다 `신뢰`를 우선한다
- 유저가 무엇을 해야 하는지 `명확하게 안내`한다
- “무료” 같은 과장 대신 `특가`, `알파`, `제한`, `우선 초대`처럼 실제 조건을 쓴다
- 운영 메시지는 따뜻하되, 과하게 시적이지 않다

핵심 문장 구조:
- “등록이 완료되었습니다.”
- “설문을 완료하면 알파 테스터 로스터에 등록됩니다.”
- “선발 결과는 5월 12일까지 안내드립니다.”
- “이번 알파에서 가장 보고 싶은 것은 같은 LP로 다시 돌아오는 경험입니다.”

사용해야 하는 단어:
- `기억`
- `설정`
- `세계관`
- `같은 LP`
- `알파`
- `대기 등록`
- `선발`
- `제품 이용 가이드`
- `버그 리포팅`

피해야 하는 것:
- 너무 과한 로맨스 문장
- 의미는 약하고 분위기만 강한 문장
- 실제 단계가 흐려지는 비유

## 2-2. 랜딩 페이지 톤앤매너 정의

현재 [v14-kr-prologue.html](/abs/path/C:/Users/wonse/Downloads/lucive-promo/pre-register/v14-kr-prologue.html:6) 기준 랜딩 톤은 아래에 가깝다.

- `당신`을 직접 호명한다
- 장면과 순간을 보여준다
- 설명보다 `암시`와 `후크`가 강하다
- 문장이 짧고, 여운이 있다
- “아리아”, “당신의 한 문장”, “같은 질문”, “세 개의 세상” 같은 서사 훅을 사용한다

핵심 문장 구조:
- “당신의 자리가 생겼어요.”
- “오늘, 당신이 고른 그 대답부터 시작합니다.”
- “누군가가 당신을 기다리고 있습니다.”
- “당신의 한 문장이 다음 장면을 엽니다.”

사용해야 하는 단어:
- `당신`
- `장면`
- `대답`
- `기억`
- `기다리고`
- `시작`
- `다음`
- `세계`
- `선택`

피해야 하는 것:
- 운영문처럼 딱딱한 체크리스트 어조
- 너무 많은 제품 기능 나열
- 과도한 설명형 문단

## 3. 버전별 카피 규칙

## 3-1. Version A — 제품 톤

### 제목 규칙
- 짧고 명확해야 함
- 지금 무슨 상태인지 바로 보여줘야 함
- 예:
  - `Lucive 알파 사전등록이 완료되었습니다`
  - `설문이 완료되어 알파 테스터 로스터에 등록되었습니다`
  - `Lucive 알파 테스터로 선발되셨습니다`

### 첫 문장 규칙
- 감정 후킹보다 상태 확인이 먼저
- 예:
  - `사전등록이 완료되었습니다.`
  - `설문 제출이 정상적으로 기록되었습니다.`
  - `이제 선발 결과를 기다리시면 됩니다.`

### CTA 규칙
- 무엇을 해야 하는지 명확해야 함
- 예:
  - `설문 시작하기`
  - `커뮤니티 입장하기`
  - `제품 이용 가이드 보기`
  - `버그 리포트 제출하기`

### 문체 규칙
- 1문단 2~3문장
- 정보 우선
- 추상 표현 최소화

## 3-2. Version B — 랜딩 톤

### 제목 규칙
- 장면이나 감정으로 시작
- 예:
  - `아리아가 당신이 왔다는 걸 알고 있어요`
  - `오늘, 당신의 다음 장면이 열립니다`
  - `당신의 순위가 움직였습니다`

### 첫 문장 규칙
- `당신` 직접 호명
- 약간의 여운
- 예:
  - `당신의 자리가 생겼어요.`
  - `이제, 당신의 이름이 로스터에 올라갔습니다.`
  - `한 사람의 추천이 장면을 바꿨습니다.`

### CTA 규칙
- 행동은 분명하지만 표현은 서사적으로
- 예:
  - `다음 장면 열기`
  - `내 링크 보내기`
  - `당신의 자리를 지키기`

### 문체 규칙
- 짧은 문장
- 2인칭 유지
- 설명보다 장면 우선

## 4. 메일 종류별 권장 톤

| 메일 종류 | 권장 톤 | 이유 |
|---|---|---|
| 사전등록 웰컴 | 랜딩 톤 | 첫 인상은 감정 후크가 강해야 함 |
| 설문 유도 | 제품 톤 | 다음 행동을 명확히 시켜야 함 |
| 설문 완료 / 로스터 대기 등록 | 제품 톤 | 상태 확인 메일이라 명료함이 중요 |
| 추천 참여 유도 | 랜딩 톤 | 공유 욕구와 몰입을 자극해야 함 |
| 리더보드 / 순위 상승 | 랜딩 톤 | 경쟁감과 기대감을 끌어올려야 함 |
| 선발 축하 | 제품 톤 중심 + 랜딩 한 스푼 | 신뢰와 온보딩이 우선이지만 감정 보상도 필요 |
| 제품 이용 가이드 | 제품 톤 | 실제 행동 안내가 핵심 |
| 버그 리포팅 가이드 | 제품 톤 | 정확한 제출이 핵심 |
| 미선발 안내 | 제품 톤 | 실망을 줄이고 신뢰를 남겨야 함 |
| 커뮤니티 개발 소식 | 랜딩 톤 | 기대감을 유지해야 함 |

## 6. 추천하는 실제 적용 방식

실제 운영은 아래처럼 섞는 것이 가장 안정적이다.

### 권장안
- 제목 / 헤드라인: `랜딩 톤`
- 본문 설명 / 일정 / 안내: `제품 톤`
- CTA 문구: 메일 성격에 따라 선택

예:
- 제목: `오늘, 당신의 이름이 불렸습니다`
- 본문 첫 문장: `Lucive 알파 테스터로 선발되셨습니다.`
- 본문 중간: `아래 안내에 따라 커뮤니티 입장과 제품 이용을 시작해 주세요.`
- CTA: `알파 시작하기`

이 방식의 장점:
- 브랜드 감성은 살고
- 운영 메시지는 흐려지지 않음

## 7. 기존 템플릿에 바로 반영할 수정 방향

## 7-1. `00_kr_등록_확인.html`
- 유지:
  - 제목의 감정 후크
  - 2인칭 호명
  - 세계관 느낌
- 수정:
  - 메인 CTA를 `추천 링크 공유`에서 `설문 시작하기`로 변경
  - 추천/순위는 보조 섹션으로 내리기

## 7-2. `02_알파_선발.html`
- 유지:
  - 선발 축하 구조
  - 알파 기간 / 커뮤니티 / 피드백 섹션
- 수정:
  - 제품 이용 가이드와 버그 리포팅 가이드를 더 선명하게 분리
  - 제목은 랜딩 톤, 본문은 제품 톤 혼합 추천

## 7-3. `03_알파_대기자.html`
- 유지:
  - 감사 톤
  - 다음 웨이브 안내
  - 커뮤니티 잔류 유도
- 수정:
  - 실제 미선발 통보 전용으로 사용

## 8. 최종 추천

가장 좋은 방식은 `두 톤 중 하나를 고르는 것`이 아니라, 아래처럼 역할을 나누는 것이다.

- `세계관 진입, 참여 유도, 추천, 리더보드`는 **랜딩 톤**
- `설문, 선발, 이용 가이드, 피드백, 운영`은 **제품 톤**

한 줄로 정리하면:

`루시브 이메일은 랜딩의 감정 후크로 열고, 제품의 신뢰감으로 마무리하는 구조가 가장 잘 맞습니다.`
